## Findings

### [MEDIUM] esbuild <=0.24.2 — dev server CORS bypass
**Category**: Dependency
**File**: server/pnpm-lock.yaml:63
**Description**: esbuild 0.21.5 is a transitive devDependency (vitest → vite → esbuild). The vulnerability allows cross-origin requests to the local esbuild dev server. Dev-only — not present in production bundles. Blast radius: developer machines running tests only.
**Recommendation**: Upgrade vitest when a release pulls in esbuild >=0.25.0. Not release-blocking.

---

### [LOW] install.sh heredoc interpolates MCP_CONFIG/SERVER_DIST as unescaped JS strings
**Category**: Shell Injection / Path Handling
**File**: install.sh:123
**Description**: Unquoted <<EOF heredoc embeds $MCP_CONFIG and $SERVER_DIST inside JS single-quoted string literals. A path containing a single quote would break JS syntax. Theoretical only — HOME and FORGE_DIR are operator-controlled and virtually never contain single quotes on macOS/Linux.
**Recommendation**: Apply `printf '%q'` to both variables before the heredoc (matching the FORGE_DIR_SAFE pattern already used for the SHEOF dispatcher), or pass via process.env.

### [LOW] curl|bash install pattern
**Category**: Supply Chain
**File**: install.sh:6
**Description**: Documented `curl -fsSL ... | bash` install method allows MITM or compromised CDN to execute arbitrary code. Standard risk profile for developer toolchains (Homebrew, Rustup). Script runs as user, not root.
**Recommendation**: Document as convenience path for trusted machines only. Recommend git clone + review for compliance environments.

### [LOW] update.sh pulls from main without tag/signature verification
**Category**: Supply Chain
**File**: update.sh:19
**Description**: `git pull origin main` then re-runs install.sh with no version pinning or signature check. Compromised origin would silently upgrade to attacker-controlled version.
**Recommendation**: Document risk. Consider `git verify-commit HEAD` or signed-tag check before proceeding.

## Accepted Risk (INFO)

| Item | Rationale |
|---|---|
| `runVerify` uses `sh -c` with plan content | Explicit trust boundary — plan files are developer-authored local files, not externally-sourced. 60s timeout, 500-byte stderr cap, CLAUDECODE guard enforced. |
| `--dangerously-skip-permissions` in spawnClaude | Required for autonomous operation. Blast radius constrained by `--allowedTools` whitelist (fail-secure to read-only for unknown types). |
| `assertNoFlagInjection` no path traversal check | Correct — values are argv elements to `spawn()`, not shell strings. No shell expansion occurs on argv arrays. |
| Top-level symlinks in `resolveScope` | Symlinks silently dropped (not followed). Safe behavior; comment clarity could be improved. |
| **No hardcoded secrets found** | Full scan of all TS, shell, and compliance files. Clean. |

## Constraints Verified

- CLAUDECODE env var deleted from child process before spawn ✓

## Verdict

Three LOW findings are all theoretical risk with no known exploitation path in normal developer environments. Recommend resolving the MEDIUM esbuild transitive dep in the next dependency update cycle.
